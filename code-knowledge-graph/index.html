Your RAG Pipeline Doesn't Understand Code Structure. Mine Uses a Knowledge Graph.

Why vector search fails on codebases, and how graph-based RAG actually works.

I asked my RAG system "how does authentication work in this codebase?" It returned five code snippets — all technically relevant, none showing the actual flow. It found the login function, the token validator, and the middleware. But it couldn't connect them into a coherent answer because it didn't understand that they form a pipeline.

I call this The Graph Gap — the difference between what vector search retrieves and what developers actually need to understand code.

Why Vector Search Fails on Code

Consider a typical authentication flow across three files:

# auth/login.py
def login(username, password):
    user = UserStore.lookup(username)
    if verify_password(password, user.hash):
        return TokenService.create_token(user)

# auth/tokens.py
class TokenService:
    def create_token(self, user):
        payload = {"sub": user.id, "exp": now() + TTL}
        return jwt.encode(payload, SECRET_KEY)

# middleware/auth.py
def require_auth(handler):
    def wrapper(request):
        token = request.headers.get("Authorization")
        user = TokenService.validate_token(token)
        return handler(request, user=user)
    return wrapper

A vector search for "how does authentication work" might return all three snippets. But it can't tell you that login() calls TokenService.create_token(), which produces a JWT that require_auth() later validates via TokenService.validate_token(). The flow — the actual answer — lives in the relationships between these functions, not in the functions themselves.

Worse, in a large codebase, the relevant snippets might be buried under dozens of higher-similarity but less relevant results. A utility function with "auth" in its docstring might rank higher than the middleware that actually enforces authentication.

The Graph Approach

I built two tools that solve this by modeling code as a knowledge graph rather than a bag of embeddings.

local-code-knowledge-graph parses your codebase and builds a graph of relationships: function calls, class inheritance, module imports, type references, and data flow. Every entity (function, class, module, variable) becomes a node. Every relationship becomes an edge.

graph-rag-private-codebases queries that graph with graph-aware RAG. When you ask "how does authentication work," it doesn't just find relevant code — it traverses the graph to reconstruct the flow. It starts at login(), follows the call to create_token(), finds every consumer of tokens, and returns a connected subgraph that tells the complete story.

What the Graph Captures

The knowledge graph indexes six relationship types that embeddings miss entirely:

  1. Call graphs — which functions call which other functions, including cross-file and cross-module calls.
  2. Import chains — module dependency trees, including transitive dependencies.
  3. Type hierarchies — class inheritance, interface implementations, type aliases.
  4. Data flow — how data moves through the system, from input to storage to output.
  5. Error propagation — which functions throw exceptions and where they're caught.
  6. Configuration dependencies — which code paths depend on environment variables, feature flags, or config files.

When you query "what happens when a user uploads a file," the graph traverses from the upload endpoint through validation, storage, thumbnail generation, and notification — returning the complete pipeline, not fragments.

The Privacy Argument

Both tools run entirely locally. Your code never leaves your machine. There's no cloud indexing, no API calls during graph construction, no telemetry.

This matters for private codebases — proprietary business logic, internal tools, pre-release features. The standard alternative is a cloud-hosted code intelligence service that requires uploading your source code. For many organizations, that's a non-starter. Compliance, IP protection, and security policies prohibit sending source code to third-party services.

Local graph construction means you get the intelligence without the exposure.

Embedding + Graph: Better Together

Graph RAG doesn't replace vector search — it augments it. The system uses a hybrid retrieval strategy:

  1. Vector search identifies candidate nodes — functions and classes that are semantically similar to the query.
  2. Graph traversal expands from those candidates — following call chains, inheritance trees, and data flow to find the connected context.
  3. Ranking combines semantic similarity with graph centrality — nodes that are both relevant AND well-connected rank highest.

This hybrid approach consistently outperforms pure vector search on structural queries ("how does X work," "what calls Y," "what happens when Z fails") while maintaining competitive performance on simple lookup queries ("find the function that validates emails").

When NOT to Use This

  • Small codebases under 10K lines of code. Grep and manual navigation are faster. The graph adds overhead that isn't justified when you can read the whole codebase in an afternoon.
  • Documentation-only RAG. If you're indexing markdown, wikis, or API docs, there are no code relationships to model. Standard vector search works fine.
  • Single-language scripting projects. If your entire project is one Python script with no imports, there's no graph to build.
  • Real-time IDE completion. Graph construction takes seconds to minutes depending on codebase size. It's designed for query-time intelligence, not keystroke-level autocomplete.

The Bigger Picture

The AI coding tool ecosystem invested heavily in making models write code faster. But writing code was never the hard part for experienced engineers — understanding existing code was. Onboarding to a new codebase, tracing bugs through unfamiliar call chains, understanding why a particular design decision was made.

Graph-based RAG addresses the understanding problem directly. Instead of asking "generate code that does X," you ask "explain how X currently works in this codebase" — and get an answer that reflects the actual architecture, not a hallucinated guess.
