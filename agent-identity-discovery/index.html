Your AI Agents Can't Find Each Other. I Built DNS for Agents.

Two tools for the same unsolved problem: agent discovery and identity in multi-agent systems.

The Problem Nobody Talks About

Eight agents, three services, one critical workflow. Agent A needed to call Agent B to validate a code review. The connection? A hardcoded URL pointing to localhost:4200.

When Agent B moved to port 4300 during a routine deploy, the pipeline silently failed. No error — just Agent A calling into the void, retrying five times, then giving up. The code review queue backed up for six hours before anyone noticed.

Adding Agent C the following week meant updating configuration files in three separate repos. Agent D required four. The coupling grew linearly with every new agent, and every deploy became a coordination tax.

This is the service discovery problem. Microservices solved it a decade ago with tools like Consul, etcd, and Kubernetes DNS. But the AI agent ecosystem has no equivalent. Agents discover each other through hardcoded endpoints, shared config files, or — worst of all — natural language instructions that say "call the code review agent at this URL."

Why Agent Identity Is Harder Than Service Discovery

Traditional service discovery assumes a controlled environment. Services register with a known orchestrator. The orchestrator manages health checks, routing, and deregistration.

Agent systems break these assumptions in three ways:

  1. Agents are autonomous. They spin up, shut down, and migrate without central coordination. A service mesh assumes stable topology. Agents assume nothing.
  1. Capabilities matter more than endpoints. When Agent A needs "someone who can review Python code," it shouldn't care whether that's Agent B on port 4200 or Agent F on a different machine entirely. Discovery should be capability-based, not address-based.
  1. Trust is unverified. When Agent B responds, how does Agent A know it's actually Agent B? In most current setups, it doesn't. It trusts the response because it came from the expected URL — the same security model as 1990s email.
MIT's 2026 agent infrastructure report called identity "the most underbuilt layer in the multi-agent ecosystem." They're right. We're building increasingly sophisticated agent networks on a foundation of implicit trust and brittle addressing.

What I Built: DNS for Agents

I built two open-source tools that work together like DNS works for the internet.

[agent-identity-registry](https://github.com/phoenix-assistant/agent-identity-registry) handles identity. Every agent gets a decentralized identifier (DID) — a cryptographically verifiable identity that doesn't depend on a central authority. The registry stores capabilities, trust chains, and verification keys.

[agent-dns-registry](https://github.com/phoenix-assistant/agent-dns-registry) handles discovery. Agents register their capabilities ("I can review Python," "I can run security scans"), and other agents query by capability rather than address. The registry resolves capability queries to the right agent, checking health and trust before returning a result.

Together, they form a complete agent addressing system:

# Register an agent with capabilities
curl -X POST http://localhost:3900/register \
  -H "Content-Type: application/json" \
  -d '{
    "agentId": "code-reviewer-01",
    "capabilities": ["python-review", "security-audit"],
    "endpoint": "http://agent-b:4200",
    "did": "did:agent:code-reviewer-01",
    "publicKey": "..."
  }'


# Discover by capability (not by address)
curl http://localhost:3900/discover?capability=python-review
# Returns: verified agent endpoint + trust chain

Bad vs. Good: Agent Communication

Without identity registry: 

# config.yaml — hardcoded, brittle, unverified
agents:
  code_reviewer: http://localhost:4200
  security_scanner: http://localhost:4300
  test_runner: http://localhost:4400
  # Move any port → everything breaks
  # No verification that responses are authentic

With identity registry: 

# config.yaml — capability-based, verified
discovery:
  registry: http://localhost:3900
  require_verification: true
  # Agents discovered by capability
  # Trust chains verified automatically
  # Health-aware routing built in

The Trust Chain Model

The identity system uses a three-tier trust model:

  1. Root of Trust — your organization's signing key. You control who can register.
  2. Agent Identity — each agent holds a DID signed by the root. Verifiable without calling home.
  3. Capability Attestation — capabilities are signed claims, not self-reported labels. An agent says "I can review Python" only if its attestation chain validates.
This plugs into existing infrastructure. If you already use OAuth or mTLS, the registry integrates as an additional verification layer rather than replacing your auth stack.

The Implementation Checklist

Copy this to evaluate whether your multi-agent system needs an identity layer:

  • [ ] More than 3 agents communicating with each other
  • [ ] Agents deployed across multiple machines or containers
  • [ ] Agent endpoints change during deploys or scaling
  • [ ] No verification that agent responses are authentic
  • [ ] Adding a new agent requires config changes in multiple places
  • [ ] Agent failures cascade because of hardcoded dependencies
If you checked three or more, you have a discovery problem. If you checked the trust items, you also have an identity problem.

When NOT to Use This

  • Single-agent systems. If you have one agent doing everything, you don't need discovery.
  • Fully trusted internal networks. If all agents run on one machine behind a firewall and you control all code, the trust layer adds overhead without proportional benefit.
  • Research prototypes. If you're experimenting and endpoints don't change, hardcoded URLs are fine. Identity infrastructure is for production multi-agent deployments.
  • Latency-critical hot paths. The registry adds a lookup hop. For sub-millisecond requirements, direct connections with cached resolution may be necessary.

The Bigger Picture

The agent ecosystem is replaying the history of distributed systems, roughly fifteen years behind. We went through the same evolution with microservices: hardcoded URLs → service registries → service meshes → zero-trust networking.

Agent systems will follow the same path. The question is whether the ecosystem builds shared infrastructure or every team reinvents discovery and identity from scratch.

These tools are a starting point. They solve the immediate problem — agents finding each other reliably and verifying identity — without requiring a complete platform migration.
