The 80% Automation Playbook — How to Build a CLI Pipeline That Writes Your PRs

Subtitle: I automated the boring 80% of my job in a weekend. The other 20% got harder.

Tags: AI, Developer Productivity, Automation, Claude, Software Engineering

I didn't plan to automate my job. I was just tired of triaging Jira tickets at 9am on a Monday.

It started with a script that pulled issues from our GitLab API. By Friday, that script was classifying tickets, spinning up branches, writing code, and opening PRs — all without me touching my keyboard. I went from 6 PRs a week to 30+. And then everything broke in a way I didn't expect.

The Problem Nobody Warns You About

Most "automate your workflow" tutorials stop at the happy path. Hook up an LLM, feed it a prompt, watch the magic. What they don't tell you: the automation isn't the hard part. Trusting the output is.

I watched an engineer on Reddit describe the same pattern I'd stumbled into. 11 years of experience, GitLab API integration, Claude CLI doing the heavy lifting. Their system pulled issues, classified them by complexity, auto-branched, and opened PRs. Worked great — until a "simple CSS fix" rewrote a shared component and broke three other pages.

The 80% of tickets that are genuinely routine? Automatable. The problem is that humans are terrible at predicting which tickets fall in that 80%.

The Ticket Triage Matrix

I call it the Ticket Triage Matrix — a classifier that runs before any automation touches your codebase. It's not about whether a ticket looks simple. It's about whether it stays simple.

The matrix scores every ticket on four dimensions:

1. Blast radius — How many files/components does this touch? (1 file = safe, 5+ = manual)

2. Test coverage — Does the affected code have tests? (>80% = safe, <50% = manual)

3. Dependency depth — Is this a leaf node or does it sit in a call chain? (leaf = safe, hub = manual)

4. Historical volatility — Has this file caused rollbacks before? (stable = safe, hot = manual)

Score 4/4? Automate it. Score 2/4? Human reviews the PR before merge. Score 0-1? Don't even try.

What This Looks Like: Bad vs Good

Bad — "just automate everything":

# dangerous-pipeline.yml
on_new_ticket:
  - classify_with_llm
  - create_branch
  - generate_code
  - open_pr
  - auto_merge  # 💀

You'll ship a breaking change by Thursday.

Good — gated automation:

# safe-pipeline.yml
on_new_ticket:
  - classify_with_llm
  - score_with_triage_matrix:
      blast_radius: check_files_touched
      test_coverage: query_codecov
      dependency_depth: parse_import_graph
      historical_volatility: git_log_rollbacks
  - gate:
      score_4: auto_branch → auto_code → auto_pr → auto_merge
      score_2_3: auto_branch → auto_code → auto_pr → HUMAN_REVIEW
      score_0_1: assign_to_human

The difference? One line: the gate. That's it. But it's the line that keeps you employed.

The Copy-Paste Starter

Here's the classifier prompt I use. Drop it into your pipeline and adapt:

## Ticket Triage Classifier

Given this ticket, score each dimension 0 or 1:

1. BLAST RADIUS: Will this change touch ≤2 files? (1=yes, 0=no)
2. TEST COVERAGE: Do affected files have >80% test coverage? (1=yes, 0=no)
3. DEPENDENCY DEPTH: Is the affected code a leaf node (not imported by 3+ modules)? (1=yes, 0=no)
4. HISTORICAL VOLATILITY: Has this file had 0 rollbacks in the last 90 days? (1=yes, 0=no)

Output: { "scores": [x,x,x,x], "total": N, "recommendation": "automate|review|manual" }

Ticket: {ticket_description}
Files likely affected: {predicted_files}

Pair this with your GitLab/GitHub API, a code coverage tool, and git log --oneline for rollback history. The whole thing runs in under 10 seconds per ticket.

When NOT to Use This

Don't automate ticket triage if:

  • Your codebase has <40% test coverage overall. You're flying blind — the matrix can't score what doesn't exist. Fix coverage first.
  • You're a team of 1-2. The overhead of building the pipeline isn't worth it until you're drowning in tickets. Just use Copilot in your editor.
  • Your tickets are vague. "Fix the thing" doesn't give the classifier enough signal. Garbage in, garbage out. Fix your ticket hygiene first.
  • You ship to regulated environments. Healthcare, finance, aviation — every PR needs a human signature regardless. The matrix helps prioritize, but it can't replace compliance review.

What Actually Happened

After three weeks with the Ticket Triage Matrix, here's what I found:

  • ~35% of our tickets scored 4/4. Those got fully automated.
  • ~40% scored 2-3. Those got auto-coded but human-reviewed. Review time dropped because the PR was already well-scoped.
  • ~25% scored 0-1. Those went straight to a human. No wasted automation cycles.

The surprise? My actual engineering work got harder. The easy tickets disappeared. Every ticket on my plate was now a gnarly, high-blast-radius problem. I got better faster because I wasn't wasting cycles on CSS tweaks and copy changes.
