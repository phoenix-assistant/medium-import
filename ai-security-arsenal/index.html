# I Shipped an AI Agent With eval() in the Tool Chain. Then I Built 5 Security Scanners. Here's the Arsenal.

Traditional security tools can't see AI vulnerabilities. I built the tools that can.

Three weeks ago I reviewed a PR for an AI agent. Clean code. Good tests. LangChain tools looked reasonable. I approved it.

Two days later, a teammate found that one of the tools called eval() on a string that originated from user input, passed through the LLM, and arrived at the tool completely unvalidated. A prompt injection away from arbitrary code execution.

My review process — the one I've used for years — completely missed it. Because my brain was trained to catch traditional vulnerabilities, not agent-specific ones. That failure launched me into building five different security tools, each targeting a different part of the AI attack surface.

The AI Security Stack

I call it The Five-Layer Defense — a framework for mapping AI-specific attack surfaces that traditional security tools can't see.

After months of building, I've identified five distinct security layers that every team deploying AI needs. Most teams cover zero of them. The best cover two. Here's all five:

Layer 1: Agent Code Scanning (The Agent Attack Surface)

Tool: [agent-security-scanner](https://github.com/phoenix-assistant/agent-security-scanner)

Static analysis for agent-specific vulnerabilities. Traditional linters catch eval() — this catches eval() fed by LLM output from user-controlled prompts. The attack chain is what matters.

[Code: # .agent-security.yml — drop in your repo rules: prompt_injection: severity: critical tool_chain_escalation: severity: high over_permissioned: severity: high missing_sandbox: severity: critical data_exfiltration: severity: critical]

Layer 2: AI-Powered Vulnerability Discovery (LARA)

Tool: [ai-vulnerability-scanner-oss](https://github.com/phoenix-assistant/ai-vulnerability-scanner-oss)

Beyond CVE databases. Uses local LLMs (Ollama/llama.cpp) to reason about your code — finding logic bugs, path traversals, and injection patterns that no CVE database knows about. Runs entirely on your machine. Your code never leaves.

The key insight: CVE databases are backward-looking. They catalog known vulnerabilities. LARA reasons forward — identifying patterns that could be vulnerable based on code structure and data flow, even if no CVE exists yet.

LARA: Local AI Risk Assessment — Lexical scan → AST analysis → Reasoning pass → Attestation.

Layer 3: Red Team Framework (The Security Blindspot)

Tool: [ai-agent-security-testing](https://github.com/phoenix-assistant/ai-agent-security-testing)

Automated red teaming for agents. Prompt injection, tool misuse, data exfiltration, privilege escalation, jailbreaking — all tested programmatically. Think OWASP ZAP, but for agents.

Layer 4: Air-Gapped Pentesting (The Air-Gap Principle)

Tool: [local-ai-pentest-suite](https://github.com/phoenix-assistant/local-ai-pentest-suite)

For security professionals who can't send client data to cloud APIs. Gemma + LangGraph + Kali, all running locally. Your client's attack surface never touches the internet.

Layer 5: Shadow AI Discovery (SHADE)

Tool: [shadow-ai-detector](https://github.com/phoenix-assistant/shadow-ai-detector)

I found 143 unauthorized AI tools in one organization. Their "approved" list had 6. SHADE scans for unauthorized AI usage: browser extensions, SaaS connections, API calls to AI endpoints your security team doesn't know about.

When NOT to Use This

  • Prototypes/hackathons — security scanning slows iteration. Ship fast, scan before production.
  • Non-agentic AI — if your AI is a simple API call with no tools, Layers 1 and 3 add overhead without value. Layer 2 still applies.
  • Fully sandboxed environments — if the agent runs in a container with no network and no persistent state, the attack surface is already constrained.

What I Learned

The hardest part wasn't building the scanners. It was accepting that my existing security instincts — honed over years of traditional application security — were almost useless for agent-specific vulnerabilities. The attack surface is different. The kill chain is different. The tools need to be different.

Start with Layer 1 (agent scanning in CI). Add layers as your deployment matures. The full stack is aspirational — but Layer 1 alone would have caught my eval() mistake.
