I Call It the Single-Agent Ceiling

Here's the pattern I hit, and I bet you've hit it too:

Month 1: You set up an AI coding agent. It's incredible. 3x productivity on boilerplate.

Month 3: You've tuned your prompts, added context files, built custom instructions. Marginal gains are shrinking. You're squeezing the last 5% out of a single-agent loop.

Month 6: You're frustrated. The agent is great at isolated tasks but terrible at coordinated work. It can write a function but can't refactor across three files without losing context. You add more context, longer prompts, more files — and hit token limits.

That's the Single-Agent Ceiling. Here's what's above it vs. below it:

❌ Below the ceiling (single-agent):
   One agent → reads everything → tries to do everything →
   loses context → produces mediocre output on complex tasks

✅ Above the ceiling (multi-agent):
   Scout agent → reads & maps the codebase (narrow scope)
   Architect agent → plans the approach (different prompt, different context)
   Coder agent → implements one piece (surgical, focused)
   Reviewer agent → validates against conventions (fresh eyes)
   → Each agent is excellent at ONE thing

The oh-my-claudecode repo proved this at scale. Teams running 3-4 specialized agents outperform teams running one "do-everything" agent — not because the models are better, but because the architecture is better.

The 4-Agent Starter Template

Here's the setup I migrated to after seeing the trending repos. It took a weekend to implement and immediately improved my output quality.

Copy this structure:

# multi-agent-setup.yaml
# Minimum viable multi-agent coding workflow

agents:
  scout:
    role: "Codebase reader and mapper"
    tools: [file_read, grep, tree]
    constraints: "NEVER write or modify files"
    output: "context-map.md — relevant files, dependencies, patterns found"

  architect:
    role: "Solution designer"
    input: "context-map.md from scout"
    tools: [file_read]  # Read-only
    constraints: "NEVER write code. Output is a plan only."
    output: "approach.md — what to change, why, in what order"

  coder:
    role: "Implementation"
    input: "approach.md from architect"
    tools: [file_read, file_write, test_run]
    constraints: "Only modify files listed in approach.md"
    output: "Changed files + test results"

  reviewer:
    role: "Quality gate"
    input: "Diff from coder + original context-map.md"
    tools: [file_read, lint, test_run]
    constraints: "NEVER modify files. Flag issues only."
    output: "review.md — pass/fail + specific issues"

flow:
  1: scout    # 30 seconds — reads and maps
  2: architect # 45 seconds — plans approach
  3: coder    # 2-5 minutes — implements
  4: reviewer  # 30 seconds — validates
  # If reviewer fails → back to step 3 with review.md as additional context

The key insight: constraints matter more than capabilities. The scout that CANNOT write code produces better context maps than an all-purpose agent. The architect that CANNOT code produces cleaner plans. Limitations create focus.

What I Got Wrong Before

I was doing what most people do — giving one agent maximum permissions and maximum context, then wondering why it produced unfocused output.

The learn-claude-code repo showed me something obvious in hindsight: the best agent harnesses are simple orchestrators over constrained specialists. Not complex prompts over general-purpose models.

My old setup: 2,000-word system prompt, 15 tools, read/write access to everything.

My new setup: 4 agents, each with a 200-word prompt and 2-3 tools.

Better output. Fewer hallucinations. Easier to debug because you know which agent failed at which step.

When This Does NOT Apply

  • Simple tasks — if the task takes < 5 minutes with a single agent, multi-agent adds overhead for no gain
  • Exploration/prototyping — when you don't know what you're building yet, a single conversational agent is faster
  • Teams without CI/automation — multi-agent assumes you can run the flow automatically. If everything is manual, stick with one agent.
  • Cost-sensitive environments — 4 agents = 4x the API calls. For hobby projects, the math doesn't work.

Multi-agent is for repeated, complex workflows where quality and consistency matter more than cost per task.
